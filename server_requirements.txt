Технические требования к серверу для развертывания сайта SQDCP
================================================================

Проект состоит из frontend-приложения на React/Vite и backend-приложения
на Flask. Сейчас backend использует SQLite, но для production-развертывания
и роста числа пользователей будет выполнен переход на PostgreSQL.

Так как возможности последующего наращивания мощности сервера, скорее всего,
не будет, сервер нужно выбирать не по минимальным требованиям, а с запасом.


1. Рекомендуемая конфигурация сервера
-------------------------------------

Основная рекомендация:

- CPU: 8 vCPU
- RAM: 16 GB
- Диск: 50 GB NVMe SSD
- ОС: Ubuntu Server 22.04 LTS или 24.04 LTS
- База данных: PostgreSQL 15/16
- Web-сервер: Nginx
- Backend runtime: Python 3.11+ с Gunicorn
- Frontend runtime: статическая сборка React-приложения через Nginx
- Backup: ежедневные резервные копии PostgreSQL и регулярные snapshot-копии
  сервера

Такой сервер лучше подходит для сценария, когда PostgreSQL, backend и frontend
будут размещены на одной машине, а увеличить ресурсы позже будет сложно или
невозможно.

Диск больше 50 GB для текущего состояния проекта, скорее всего, не нужен.
Приложение не хранит пользовательские файлы и большие вложения, поэтому диск
будет расходоваться в основном на операционную систему, зависимости, PostgreSQL,
логи и временное хранение свежих резервных копий. При регулярной выгрузке
backup во внешнее хранилище 50 GB NVMe SSD должно быть достаточно.

Количество пользователей около 20 почти не влияет на объем диска: учетные
записи, задачи, доски и отделы занимают очень мало места. Основной расход диска
создают не пользователи, а операционная система, PostgreSQL, индексы, логи,
резервные копии и запас под обновления.

Оценка дискового пространства:

- 25-30 GB: технически достаточно, чтобы запустить проект, но почти без запаса;
- 40 GB: минимально разумный вариант для production без хранения backup на
  сервере;
- 50 GB: рекомендуемый вариант для production с небольшим запасом;
- 80-100 GB: имеет смысл только если резервные копии будут длительно храниться
  прямо на сервере или если позже появится хранение файлов пользователей.


2. Минимально допустимая production-конфигурация
------------------------------------------------

Если бюджет ограничен, минимально допустимый вариант:

- CPU: 4 vCPU
- RAM: 16 GB
- Диск: 40-50 GB NVMe SSD
- ОС: Ubuntu Server 22.04 LTS или 24.04 LTS
- База данных: PostgreSQL 15/16

Варианты с 2 vCPU и 4-8 GB RAM лучше рассматривать только для тестового
стенда или пилотного запуска. Для production-сервера без возможности апгрейда
такого запаса может быстро не хватить.


3. Почему нужен запас
---------------------

Основная нагрузка проекта приходится на:

- работу Flask backend;
- запросы к PostgreSQL;
- авторизацию пользователей;
- чтение и изменение досок, отделов, проектов и задач;
- хранение истории данных;
- резервное копирование;
- обслуживание одновременных пользователей.

Даже если сайт сейчас не выглядит тяжелым, с ростом числа пользователей будет
расти количество записей в базе данных, количество одновременных запросов и
объем резервных копий. PostgreSQL также требует оперативной памяти для
кеширования данных и стабильной работы под нагрузкой.


4. Рекомендуемая схема размещения
---------------------------------

На одном сервере:

- Nginx принимает HTTPS-запросы;
- Nginx отдает собранный frontend из папки dist;
- Nginx проксирует API-запросы на Flask backend;
- Flask backend запускается через Gunicorn;
- PostgreSQL работает локально на этом же сервере;
- резервные копии БД сохраняются локально и дополнительно выгружаются
  во внешнее хранилище.

Node.js на сервере нужен только для сборки frontend. В production frontend
не должен запускаться через Vite dev server.


5. Что обязательно подготовить перед production-запуском
--------------------------------------------------------

- Перевести базу данных с SQLite на PostgreSQL.
- Вынести строку подключения к БД в переменную окружения DATABASE_URL.
- Вынести JWT_SECRET_KEY и другие секреты из кода в переменные окружения.
- Отключить Flask debug mode.
- Запускать backend через Gunicorn, а не через app.run().
- Настроить Nginx и HTTPS-сертификат.
- Настроить firewall: открыть только SSH, HTTP и HTTPS.
- Настроить автоматический restart backend-сервиса через systemd.
- Настроить ежедневные backup PostgreSQL.
- Проверять восстановление из backup хотя бы перед запуском и после крупных
  изменений.
- Настроить ротацию логов.
- Ограничить доступ к PostgreSQL только локальным подключением или защищенной
  внутренней сетью.


6. Резервное копирование
------------------------

Рекомендуемая схема:

- ежедневный pg_dump базы данных;
- хранение минимум 14-30 последних ежедневных копий;
- еженедельный полный snapshot сервера;
- копирование backup во внешнее хранилище, не только на этот же сервер;
- периодическая проверка восстановления backup на отдельной машине или
  тестовом окружении.

Важно: backup, который хранится только на основном сервере, не защищает от
потери сервера целиком.


7. Итоговая рекомендация
------------------------

Если сервер выбирается один раз и без надежной возможности последующего
увеличения ресурсов, оптимальный выбор:

8 vCPU, 16 GB RAM, 50 GB NVMe SSD, Ubuntu Server LTS, PostgreSQL 15/16.

Это даст запас для роста пользователей, увеличения базы данных, резервного
копирования и более стабильной работы приложения в production.

Если бюджет не позволяет выбрать такую конфигурацию, нижняя граница для
production без дальнейшего апгрейда:

4 vCPU, 16 GB RAM, 40-50 GB NVMe SSD.

Ниже этой конфигурации сервер лучше использовать только как тестовый или
временный пилотный стенд.
